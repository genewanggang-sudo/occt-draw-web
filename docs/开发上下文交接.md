## 当前产品方向

项目目标是三维云端 CAD 工作台，不再做二维出图编辑器。

已确认方向：

- 页面：React
- 构建：Vite
- 包管理：pnpm workspace
- Node/pnpm 管理：mise，仅对当前项目生效
- 渲染：原生 WebGL，不引入 Three.js
- 几何内核：未来通过 `occt-draw-core` Wasm 提供
- 命令、选择、草图、参数化编辑状态：TypeScript 实现
- 文档、产品文案和工程文档：中文

## 当前阶段

当前仍处于 CAD 工作台基础能力收口阶段，重点是让 `math / core / display / renderer / renderer-webgl / editor` 的边界稳定，再继续推进草图系统 v1。

## 最近架构收口

最近完成的基础能力：

- `packages/editor` 已落地，避免命令逻辑散落在 React 组件中。
- `display` 作为投影层，负责 `CadDocument + EditDraft -> DisplayModel`。
- `renderer` 定义相机、包围盒、拾取和渲染输入协议。
- `renderer-webgl` 只消费 `DisplayModel` 并执行 WebGL 绘制。
- 默认场景已移除调试立方体、坐标轴、地面网格，只保留基准面和原点。
- 原点已作为 `core` 中正式 `reference-origin` 对象存在，渲染为 origin marker。
- 基准面已投影为半透明面片、边框线和文字 label。
- `math.Plane` 已从 `origin + normal` 升级为 `origin + normal + xAxis + yAxis` 的无限工作平面。
- 默认相机和标准视图已调整为 CAD 常见的 Z-up 语义：Top 是 XOY/XY 平面，Front 是 XZ/ZX 方向，Right 是 YZ 方向。

最近完整校验通过：

```bash
pnpm check
```

## 工作平面当前状态

`math.Plane` 当前是类实例，不再只是一个无限数学平面的最小表达。

当前属性：

- `origin: Point3`
- `normal: Vec3`
- `xAxis: Vec3`
- `yAxis: Vec3`

构造入口：

- `new Plane(origin, normal, xAxis?)`
- `createPlane(origin, normal, xAxis?)`

构造时会做统一正交化：

- `normal` 归一化。
- `xAxis` 投影到平面内再归一化。
- `yAxis = normal.cross(xAxis)`。
- 非法 normal、零向量、平行 xAxis 会回退到稳定默认轴。

当前方法：

- `signedDistanceToPoint`
- `projectPoint`
- `intersectRay`
- `localToWorld`
- `worldToLocal`
- `projectPointToLocal`

兼容函数仍保留：

- `signedDistanceToPlane`
- `projectPointToPlane`
- `intersectRayWithPlane`

CAD 对象层：

- `ReferencePlaneObject` 仍然是文档对象，包含 `origin / normal / xAxis / size / planeKind`。
- `size` 只表达显示面片尺寸，不进入 `math.Plane`。
- `core.referencePlaneToPlane(object)` 负责把 `ReferencePlaneObject` 转成标准 `Plane`。

## 默认基准面当前状态

当前默认基准面仍是 `ReferencePlaneObject`，不是直接把 `Plane` 存进文档。

语义约定：

- Top：XY / XOY 平面。
- Front：ZX / XOZ 平面。
- Right：YZ / YOZ 平面。

`planeKind` 暂时保留，用于名称、默认文案和兼容；新的坐标转换应优先使用 `Plane.localToWorld / worldToLocal`，不要继续按 `planeKind` 到处硬编码坐标映射。

## Text Label 当前状态

基准面文字当前目标是接近 Onshape 的效果：

- 使用 WebGL 贴图文字，不使用 DOM overlay。
- 文字跟随基准面旋转。
- 文字参与 depth test，可被前方几何遮挡。
- 文字屏幕大小基本固定，不随模型缩放变大变小。
- 文案保持英文：`Top / Front / Right`。

当前实现链路：

```txt
reference-plane
  -> display label-batch
  -> renderer-webgl labelAtlas 生成 Canvas2D 文字贴图
  -> labelGeometry 生成 3D textured quad
  -> label shader 绘制贴图文字
```

当前 label 模型：

```txt
frame + insert + justify + heightPixels + paddingPixels + fontWeight
```

语义：

- `frame`：文字所在局部坐标系，包含 `origin / xAxis / yAxis`，都是世界坐标语义。
- `insert`：文字在 frame 内的插入点，单位是局部世界单位，不是像素。
- `justify`：文字如何挂到插入点上，包含水平、垂直和 baseline 语义。
- `heightPixels`：文字屏幕高度，用于保持视觉字号稳定。
- `paddingPixels`：文字盒子沿 frame 轴向的屏幕像素内边距，会在渲染时转成世界单位。
- `fontWeight`：字体粗细，当前支持 `100..900` 的数值字重；切换字重会按需重建 atlas，不预生成所有字重。

当前默认基准面 label：

- `fontWeight: 400`
- `heightPixels: 15`
- `paddingPixels: { x: 6, y: 6 }`
- label frame 放在基准面左上角，`yAxis` 为反向，使 `vertical: 'top'` 更符合贴角阅读语义。

当前已经处理过的点：

- 不再使用无语义的 `anchor + offsetPixels` 模型。
- 已支持文字 padding。
- 已支持传入 fontWeight。
- 文字字号已调小，字重已调细。
- atlas 不预生成所有字重，当前按显示模型中出现的字重按需重建。

## 相机导航当前状态

默认相机和标准视图已切到 Z-up，但鼠标导航仍有需要重构的地方。

当前已确认的问题：

- 缩小后旋转，有时看起来像被相机裁剪。
- 平移后再缩小、再旋转，若简单改 pivot 或 target，会引入画面跳变。
- 这些问题不是单纯调大 `near/far` 能彻底解决的，根因是 `camera.target`、`pivot`、`sceneCenter` 的职责没有完全拆清楚。

当前代码状态：

- `ViewNavigationState` 只有 `camera / drag / pivot / sceneRadius / viewportSize`。
- 平移会移动 `camera.position / camera.target`，也会移动 `pivot`。
- 缩放会为了保持鼠标下世界点稳定而移动 `camera.position / camera.target`，也会移动 `pivot`。
- 旋转时 `rotateCameraByScreenAxes` 会让 `position` 绕 `pivot` 转，并把 `target` 设置为 `pivot`。

近期做过实验但已丢弃：

- 曾尝试增加 `sceneCenter` 并按真实模型中心算 `near/far`。
- 曾尝试让缩放不移动 `pivot`。
- 曾尝试让 `position` 和 `target` 一起绕 `pivot` 转。
- 这些实验没有提交，当前代码没有保留这些改动。

下一次处理相机导航时建议先定设计，不要继续零散补丁：

1. 明确定义 `camera.target`：当前视图中心。
2. 明确定义 `orbitPivot`：旋转中心。
3. 明确定义 `sceneCenter / sceneRadius`：只用于裁剪和 fit，不直接表达用户交互中心。
4. 平移、滚轮缩放、旋转三种操作必须保持连续，不应在旋转第一帧把视图中心强行拉回模型中心。
5. 裁剪范围应覆盖真实 display bounds，而不是只依赖当前 `target` 或临时 pivot。
6. 设计确认后再改 `ViewNavigationState` 和 `rotateCameraByScreenAxes`。

## 关键架构理解

### EditDraft

`EditDraft` 是命令执行过程中的临时编辑层。

它不是 UI，也不是 renderer。它用于承载：

- 鼠标移动过程中的临时线段
- 拖动过程中的临时位置
- 可取消的草图预览
- 后续约束求解过程中的临时 working document

标准规则：

- `pointerDown` 创建或初始化 `EditDraft`
- `pointerMove` 更新 `EditDraft`
- `pointerUp` 生成正式 `DocumentOperation`
- `Esc` 丢弃 `EditDraft`

### Operation

`DocumentOperation` 不是 Request。

```txt
Request/Input：用户刚刚做了什么输入
Command：当前命令如何解释输入
EditDraft：命令过程中的临时数据
Operation：最终写入 CAD 文档的正式业务动作
Transaction：一次原子提交
TransactionGroup：一个用户动作包含多次提交
```

拖动 300 次不应该产生 300 条历史。拖动过程更新 `EditDraft`，确认后只提交一次 `Operation` 或 `TransactionGroup`。

### Display

`display` 的职责只有一个：把正式文档和临时编辑层转换为可渲染数据。

```txt
CadDocument + EditDraft -> DisplayModel -> renderer -> renderer-webgl
```

`display` 不知道 React、不知道 WebGL、不知道 CommandManager，也不修改 `CadDocument`。

### Editor

`editor` 是编辑器运行时包。

它不负责 UI，也不是 CAD 文档模型。它承接 UI 之外的编辑器交互能力：

- 当前命令和命令生命周期
- 鼠标、键盘、视图输入解释
- 选择状态和命令状态流转
- `EditDraft` 创建、更新和取消
- `DocumentOperation / DocumentTransaction` 提交入口

目标是让 `apps/web` 只负责产品装配、界面展示和事件转发。

## 当前不要做的事

- 不要回到二维出图方向。
- 不要先做复杂 UI。
- 不要引入 Three.js 或 gl-matrix。
- 不要把命令逻辑写进 React 组件。
- 不要让 `renderer-webgl` 反向知道 core、command 或 sketch。
- 不要在鼠标移动过程中频繁写正式文档历史。
- 不要绕过 `EditDraft / Operation / display` 边界。
- 不要默认启动子 Agent，除非用户明确要求。
- 当前阶段不主动新增单元测试；用户更倾向浏览器手动验收，但 `pnpm check` 必须通过。

## 下一步推荐

优先处理相机导航设计和实现，再继续推进草图系统 v1。

相机导航推荐顺序：

1. 先写清楚 `target / orbitPivot / sceneCenter` 的职责。
2. 更新 `ViewNavigationState`，避免一个 `pivot` 同时承担视图中心、旋转中心和裁剪中心。
3. 让 pan、zoom、rotate 都保持连续。
4. 用 display bounds 计算稳定裁剪面。
5. 浏览器手动验证：平移、缩小、旋转、极端缩放、标准视图、适配视图。

草图系统 v1 推荐顺序：

1. 基于当前 `ReferencePlaneObject -> Plane` 的链路继续收口草图平面。
2. 定义最小草图数据模型：`Sketch / SketchPoint / SketchLine`。
3. 在 `editor` 中继续完善 `SketchLineCommand`。
4. 第一次点击记录起点到 `EditDraft`。
5. 鼠标移动时用 `EditDraft` 显示临时线段。
6. 第二次点击生成正式草图线。
7. `DocumentTransaction` 提交到 `CadDocument`。
8. `display` 将正式草图线通过 `Plane.localToWorld` 投影为 line batch。
9. WebGL 显示正式草图线。

第一版不做：

- 圆、圆弧、多段线
- 尺寸标注
- 约束求解
- 拉伸实体
- Wasm 调用
- 云端保存
