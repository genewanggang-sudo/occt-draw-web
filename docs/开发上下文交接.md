## 当前产品方向

项目目标是三维云端 CAD 工作台，不再做二维出图编辑器。

已确认方向：

- 页面：React
- 构建：Vite
- 包管理：pnpm workspace
- Node/pnpm 管理：mise，仅对当前项目生效
- 渲染：原生 WebGL，不引入 Three.js
- 几何内核：未来通过 `occt-draw-core` Wasm 提供
- 命令、选择、草图、参数化编辑状态：TypeScript 实现
- 文档、产品文案和工程文档：中文

## 最近架构收口

最近完成或正在完成的基础能力：

- `packages/editor` 已落地，避免命令逻辑散落在 React 组件中。
- `display` 作为投影层，负责 `CadDocument + EditDraft -> DisplayModel`。
- `renderer` 定义相机、包围盒、拾取和渲染输入协议。
- `renderer-webgl` 只消费 `DisplayModel` 并执行 WebGL 绘制。
- 默认场景已移除调试立方体、坐标轴、地面网格，只保留基准面和原点。
- 原点已作为 `core` 中正式 `reference-origin` 对象存在，渲染为 origin marker。
- 基准面已投影为半透明面片、边框线和文字 label。

最近完整校验通过：

```bash
pnpm check
```

## Text Label 当前状态

基准面文字当前目标是接近 Onshape 的效果：

- 使用 WebGL 贴图文字，不使用 DOM overlay。
- 文字跟随基准面旋转。
- 文字参与 depth test，可被前方几何遮挡。
- 文字屏幕大小基本固定，不随模型缩放变大变小。
- 文案保持英文：`Top / Front / Right`。

当前实现链路：

```txt
reference-plane
  -> display label-batch
  -> renderer-webgl labelAtlas 生成 Canvas2D 文字贴图
  -> labelGeometry 生成 3D textured quad
  -> label shader 绘制贴图文字
```

当前 label 模型已从临时的 `anchor + offsetPixels` 改为 CAD 风格的：

```txt
frame + insert + justify + heightPixels
```

语义：

- `frame`：文字所在局部坐标系，包含 `origin / xAxis / yAxis`。
- `insert`：文字在 frame 内的插入点，使用局部世界单位。
- `justify`：文字如何挂到插入点上，包含水平、垂直和 baseline 语义。
- `heightPixels`：文字屏幕高度，用于保持视觉字号稳定。

当前关键问题：

- 放大后文字仍会出现“不贴平面”或“超出平面”的观感。
- 原因之一是此前 label 为避免 z-fighting 使用了 `normal * 0.01` 的世界单位偏移。
- 放大时这个 normal 偏移会被放大成明显屏幕距离，导致文字像悬浮在平面外。
- `insert = { x: 0, y: 0 }` 会让文字盒子直接贴在角点，字形抗锯齿或 bearing 可能越过边框。

下一步推荐：

1. 不再用几何上的 `normal * 0.01` 把文字顶出平面。
2. 让文字 quad 和基准面共面或近似共面。
3. 用渲染层 depth bias / polygon offset / depth function 策略解决 z-fighting。
4. 如果需要 Onshape 那种贴角但略微内缩的观感，优先使用明确的 `insert` 局部世界单位内缩；不要重新引入无语义的 `offsetPixels`。
5. 若后续需要 UI overlay 风格的固定屏幕内缩，再单独增加 `screenOffsetPixels`，不要混入当前 CAD label 坐标模型。

## 关键架构理解

### EditDraft

`EditDraft` 是命令执行过程中的临时编辑层。

它不是 UI，也不是 renderer。它用于承载：

- 鼠标移动过程中的临时线段
- 拖动过程中的临时位置
- 可取消的草图预览
- 后续约束求解过程中的临时 working document

标准规则：

- `pointerDown` 创建或初始化 `EditDraft`
- `pointerMove` 更新 `EditDraft`
- `pointerUp` 生成正式 `DocumentOperation`
- `Esc` 丢弃 `EditDraft`

### Operation

`DocumentOperation` 不是 Request。

```txt
Request/Input：用户刚刚做了什么输入
Command：当前命令如何解释输入
EditDraft：命令过程中的临时数据
Operation：最终写入 CAD 文档的正式业务动作
Transaction：一次原子提交
TransactionGroup：一个用户动作包含多次提交
```

拖动 300 次不应该产生 300 条历史。拖动过程更新 `EditDraft`，确认后只提交一次 `Operation` 或 `TransactionGroup`。

### Display

`display` 的职责只有一个：把正式文档和临时编辑层转换为可渲染数据。

```txt
CadDocument + EditDraft -> DisplayModel -> renderer -> renderer-webgl
```

`display` 不知道 React、不知道 WebGL、不知道 CommandManager，也不修改 `CadDocument`。

### Editor

`editor` 是编辑器运行时包。

它不负责 UI，也不是 CAD 文档模型。它承接 UI 之外的编辑器交互能力：

- 当前命令和命令生命周期
- 鼠标、键盘、视图输入解释
- 选择状态和命令状态流转
- `EditDraft` 创建、更新和取消
- `DocumentOperation / DocumentTransaction` 提交入口

目标是让 `apps/web` 只负责产品装配、界面展示和事件转发。

## 当前不要做的事

- 不要回到二维出图方向。
- 不要先做复杂 UI。
- 不要引入 Three.js 或 gl-matrix。
- 不要把命令逻辑写进 React 组件。
- 不要让 `renderer-webgl` 反向知道 core、command 或 sketch。
- 不要在鼠标移动过程中频繁写正式文档历史。
- 不要绕过 `EditDraft / Operation / display` 边界。
- 不要默认启动子 Agent，除非用户明确要求。
- 当前阶段不主动新增单元测试；用户更倾向浏览器手动验收，但 `pnpm check` 必须通过。

## 下一步推荐

优先继续收口 text label 显示模型，然后再进入草图系统 v1。

text label 下一步：

1. 移除 label 的 normal 世界偏移。
2. 增加 label 渲染深度偏移策略，保证共面文字不闪烁。
3. 浏览器缩放验证：文字是否贴平面、是否超边、是否随面旋转。
4. 若效果稳定，再提交这一批 label/display/renderer 改动。

草图系统 v1 推荐顺序：

1. 定义默认草图平面和草图坐标系，第一版使用 XY 平面。
2. 定义最小草图数据模型：`Sketch / SketchPoint / SketchLine`。
3. 在 `editor` 中新增 `SketchLineCommand`。
4. 第一次点击记录起点到 `EditDraft`。
5. 鼠标移动时用 `EditDraft` 显示临时线段。
6. 第二次点击生成 `AddSketchLineOperation`。
7. `DocumentTransaction` 提交到 `CadDocument`。
8. `display` 将正式草图线投影为 line batch。
9. WebGL 显示正式草图线。

第一版不做：

- 圆、圆弧、多段线
- 尺寸标注
- 约束求解
- 拉伸实体
- Wasm 调用
- 云端保存

## 用户偏好和工作方式

- 要工业级产品思路，不接受临时堆砌方案。
- 代码和文档要能解释清楚架构意图。
- 文档使用中文。
- UI 当前只保留必要入口，避免发散。
- 出方案时优先使用架构图、流程图和简洁说明。
- 实现后要说明做了什么、是否解决之前确认的问题。
- 当前不希望默认使用多个子线程并行开发。
- 重要改动后至少跑 `pnpm check`。
