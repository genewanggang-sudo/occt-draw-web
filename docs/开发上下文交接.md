## 当前产品方向

项目目标是三维云端 CAD 工作台，不再做二维出图编辑器。

已确认方向：

- 页面：React
- 构建：Vite
- 包管理：pnpm workspace
- Node/pnpm 管理：mise，仅对当前项目生效
- 渲染：原生 WebGL，不引入 Three.js

- 文档、产品文案和工程文档：中文

## 工作平面当前状态

`math.Plane` 当前是类实例，不再只是一个无限数学平面的最小表达。

当前属性：

- `origin: Point3`
- `normal: Vec3`
- `xAxis: Vec3`
- `yAxis: Vec3`

构造入口：

- `new Plane(origin, normal, xAxis?)`
- `createPlane(origin, normal, xAxis?)`

构造时会做统一正交化：

- `normal` 归一化。
- `xAxis` 投影到平面内再归一化。
- `yAxis = normal.cross(xAxis)`。
- 非法 normal、零向量、平行 xAxis 会回退到稳定默认轴。

当前方法：

- `signedDistanceToPoint`
- `projectPoint`
- `intersectRay`
- `localToWorld`
- `worldToLocal`
- `projectPointToLocal`

兼容函数仍保留：

- `signedDistanceToPlane`
- `projectPointToPlane`
- `intersectRayWithPlane`

CAD 对象层：

- `ReferencePlaneObject` 仍然是文档对象，包含 `origin / normal / xAxis / size / planeKind`。
- `size` 只表达显示面片尺寸，不进入 `math.Plane`。
- `core.referencePlaneToPlane(object)` 负责把 `ReferencePlaneObject` 转成标准 `Plane`。

## 默认基准面当前状态

当前默认基准面仍是 `ReferencePlaneObject`，不是直接把 `Plane` 存进文档。

语义约定：

- Top：XY / XOY 平面。
- Front：ZX / XOZ 平面。
- Right：YZ / YOZ 平面。

`planeKind` 暂时保留，用于名称、默认文案和兼容；新的坐标转换应优先使用 `Plane.localToWorld / worldToLocal`，不要继续按 `planeKind` 到处硬编码坐标映射。

## Text Label 当前状态

基准面文字当前目标是接近 Onshape 的效果：

- 使用 WebGL 贴图文字，不使用 DOM overlay。
- 文字跟随基准面旋转。
- 文字参与 depth test，可被前方几何遮挡。
- 文字屏幕大小基本固定，不随模型缩放变大变小。
- 文案保持英文：`Top / Front / Right`。

当前实现链路：

```txt
reference-plane
  -> display label-batch
  -> renderer-webgl labelAtlas 生成 Canvas2D 文字贴图
  -> labelGeometry 生成 3D textured quad
  -> label shader 绘制贴图文字
```

当前 label 模型：

```txt
frame + insert + justify + heightPixels + paddingPixels + fontWeight
```

语义：

- `frame`：文字所在局部坐标系，包含 `origin / xAxis / yAxis`，都是世界坐标语义。
- `insert`：文字在 frame 内的插入点，单位是局部世界单位，不是像素。
- `justify`：文字如何挂到插入点上，包含水平、垂直和 baseline 语义。
- `heightPixels`：文字屏幕高度，用于保持视觉字号稳定。
- `paddingPixels`：文字盒子沿 frame 轴向的屏幕像素内边距，会在渲染时转成世界单位。
- `fontWeight`：字体粗细，当前支持 `100..900` 的数值字重；切换字重会按需重建 atlas，不预生成所有字重。

当前默认基准面 label：

- `fontWeight: 400`
- `heightPixels: 15`
- `paddingPixels: { x: 6, y: 6 }`
- label frame 放在基准面左上角，`yAxis` 为反向，使 `vertical: 'top'` 更符合贴角阅读语义。

当前已经处理过的点：

- 不再使用无语义的 `anchor + offsetPixels` 模型。
- 已支持文字 padding。
- 已支持传入 fontWeight。
- 文字字号已调小，字重已调细。
- atlas 不预生成所有字重，当前按显示模型中出现的字重按需重建。

## 相机导航当前状态

默认相机和标准视图已切到 Z-up，但鼠标导航仍有需要重构的地方。

当前已确认的问题：

- 缩小后旋转，有时看起来像被相机裁剪。
- 平移后再缩小、再旋转，若简单改 pivot 或 target，会引入画面跳变。
- 这些问题不是单纯调大 `near/far` 能彻底解决的，根因是 `camera.target`、`pivot`、`sceneCenter` 的职责没有完全拆清楚。

当前代码状态：

- `ViewNavigationState` 只有 `camera / drag / pivot / sceneRadius / viewportSize`。
- 平移会移动 `camera.position / camera.target`，也会移动 `pivot`。
- 缩放会为了保持鼠标下世界点稳定而移动 `camera.position / camera.target`，也会移动 `pivot`。
- 旋转时 `rotateCameraByScreenAxes` 会让 `position` 绕 `pivot` 转，并把 `target` 设置为 `pivot`。

近期做过实验但已丢弃：

- 曾尝试增加 `sceneCenter` 并按真实模型中心算 `near/far`。
- 曾尝试让缩放不移动 `pivot`。
- 曾尝试让 `position` 和 `target` 一起绕 `pivot` 转。
- 这些实验没有提交，当前代码没有保留这些改动。

下一次处理相机导航时建议先定设计，不要继续零散补丁：

1. 明确定义 `camera.target`：当前视图中心。
2. 明确定义 `orbitPivot`：旋转中心。
3. 明确定义 `sceneCenter / sceneRadius`：只用于裁剪和 fit，不直接表达用户交互中心。
4. 平移、滚轮缩放、旋转三种操作必须保持连续，不应在旋转第一帧把视图中心强行拉回模型中心。
5. 裁剪范围应覆盖真实 display bounds，而不是只依赖当前 `target` 或临时 pivot。
6. 设计确认后再改 `ViewNavigationState` 和 `rotateCameraByScreenAxes`。

## 关键架构理解

### EditDraft

`EditDraft` 是命令执行过程中的临时编辑层。

它不是 UI，也不是 renderer。它用于承载：

- 鼠标移动过程中的临时线段
- 拖动过程中的临时位置
- 可取消的草图预览
- 后续约束求解过程中的临时 working document

标准规则：

- `pointerDown` 创建或初始化 `EditDraft`
- `pointerMove` 更新 `EditDraft`
- `pointerUp` 生成正式 `DocumentOperation`
- `Esc` 丢弃 `EditDraft`

### Operation

`DocumentOperation` 不是 Request。

```txt
Request/Input：用户刚刚做了什么输入
Command：当前命令如何解释输入
EditDraft：命令过程中的临时数据
Operation：最终写入 CAD 文档的正式业务动作
Transaction：一次原子提交
TransactionGroup：一个用户动作包含多次提交
```

拖动 300 次不应该产生 300 条历史。拖动过程更新 `EditDraft`，确认后只提交一次 `Operation` 或 `TransactionGroup`。

### Display

`display` 的职责只有一个：把正式文档和临时编辑层转换为可渲染数据。

```txt
CadDocument + EditDraft -> DisplayModel -> renderer -> renderer-webgl
```

`display` 不知道 React、不知道 WebGL、不知道 CommandManager，也不修改 `CadDocument`。

### Editor

`editor` 是编辑器运行时包。

它不负责 UI，也不是 CAD 文档模型。它承接 UI 之外的编辑器交互能力：

- 当前命令和命令生命周期
- 鼠标、键盘、视图输入解释
- 选择状态和命令状态流转
- `EditDraft` 创建、更新和取消
- `DocumentOperation / DocumentTransaction` 提交入口

目标是让 `apps/web` 只负责产品装配、界面展示和事件转发。

### 已确认的相机导航设计

- 普通 rotate 不使用固定世界 Z 轴；水平旋转绕当前 `camera.up`，垂直旋转绕当前 `camera.right`。
- `pointerDown` 时解析一次 `rotateOrPanCenter`，drag 过程中固定使用该中心。
- 不使用长期累积的旧 `orbitPivot` 作为 rotate begin 的默认来源。
- `camera.target` 表示当前视图中心语义；`orbitPivot` 表示当前 rotate/pan 操作中心；`sceneCenter/sceneRadius` 用于 fit、clipping 和 fallback。
- rotate 时 `position` 和 `target` 一起围绕 `orbitPivot` 旋转，避免第一帧把视图中心强行拉回 pivot。
- pan 使用 rotate/pan center 所在的 view-normal plane，保持 pan 和 rotate 中心语义一致。
- wheel zoom 使用本次鼠标位置解析出的 `zoomAnchor` 保持鼠标下内容稳定，但不写回长期 `orbitPivot`。
- pan / rotate / zoom / fit / standard view 后都会重新按当前 camera forward 和 navigation bounds 刷新 `near/far`。

### 当前 orbit center 解析顺序

当前实现接近 Onshape 的普通 free orbit 结构：

1. 鼠标附近矩形先做 `model` depth sampling，命中则取离鼠标最近的 `worldPoint`。
2. 鼠标附近没有 model 样本时，再允许 `reference-plane`，取离鼠标最近的 `worldPoint`。
3. 鼠标附近仍无样本时，对整个 viewport 做稀疏 depth sampling，先 `includePlanes=false`。
4. model 窗口样本不足时，再 `includePlanes=true`。
5. 窗口样本足够时，对 `worldPoint` 求平均作为中心。
6. 仍失败时使用 navigation bounds center。
7. bounds 不适合时，用鼠标射线按 bounds center 深度计算 fallback 点。
8. 最后兜底 `camera.target`。

当前阈值：

- 鼠标附近采样半径：`12px`
- 窗口稀疏采样步长：`72px`
- 窗口样本最少数量：`3`
- bounds viewport fit factor：`2`

这些阈值是当前版本的工程默认值，后续可以基于手感和模型规模继续调参。

### WebGL depth sampling 当前实现

当前不是遍历 `DisplayModel` 做软件 raycast，而是 renderer-webgl 内部的 screen-space depth pass。

实现链路：

```txt
editor resolveNavigationCenter
  -> context.sampleNavigationDepths(...)
  -> CadRenderer.sampleNavigationDepths(...)
  -> renderer-webgl offscreen navigation depth pass
  -> gl.readPixels(...)
  -> canvasDepthToWorld(...)
  -> NavigationDepthSample[]
```

关键实现点：

- 当前已切换为 WebGL2 后端，初始化时直接请求 `webgl2` context，不再回退到 WebGL1。
- renderer 内部类型已切到 `WebGL2RenderingContext`。
- 主渲染 shader、label shader 和 navigation depth shader 已切到 GLSL 300 ES。
- offscreen navigation depth framebuffer 当前使用 RGBA color texture + `DEPTH_COMPONENT32F` depth texture。
- depth texture 负责真实遮挡；RGBA texture 仍负责把最终可见 fragment 的 depth 和 navigation role 暴露给 CPU。
- shader 把 `gl_FragCoord.z` 编码到 RGB，把 navigation role 编码到 alpha。
- CPU 侧仍从 RGBA color attachment 读取编码深度，不直接用 `readPixels(DEPTH_COMPONENT, FLOAT)` 计算旋转中心，以保持交互语义稳定。
- `model` 和 `reference-plane` 会进入 navigation depth pass。
- `annotation` / label 永远不参与 navigation depth pass，也不参与 navigation bounds。
- `canvasDepthToWorld(...)` 根据 `{ canvasPoint, depth01 }` 反投影出 `worldPoint`，同时计算 `viewDepth`。
- `includePlanes=false` 时只返回 model；`includePlanes=true` 时返回 model + reference-plane。
- 主渲染、label 渲染和 navigation depth pass 都已使用 WebGL2 VAO 记录固定 attribute layout，绘制时只绑定对应 VAO。

这套 depth readback 仍沿用早期 RGBA 编码设计，切换到 WebGL2 后暂时保留以降低行为变化风险。后续可以把 RGBA depth encode 改成 WebGL2-native depth readback / 更完整的 pick-depth buffer 架构，但当前不作为本轮迁移必做项。

### WebGL2 改造当前状态

本轮 WebGL2 改造已阶段性完成并验收：

- `createWebglRenderer(...)` 只请求 `webgl2` context。
- renderer-webgl 内部上下文类型使用 `WebGL2RenderingContext`。
- 主渲染、label 渲染和 navigation depth shader 都已升级到 GLSL 300 ES。
- navigation depth target 已从 depth renderbuffer 改为 depth texture。
- 主渲染、label 渲染和 navigation depth pass 都已 VAO 化。
- 刷新页面时首帧不再用 `1x1` viewport 渲染，避免 canvas 被 CSS 拉伸后出现整块蓝色闪烁。

暂不继续追加以下 WebGL2 增强项：

- MRT
- UBO
- query / timer query
- instancing
- framebuffer fallback
- render pass 大抽象

### 当前和 Onshape 仍有差距

已接近的部分：

- rotate/pan begin 时解析一次 center。
- drag 期间固定 center。
- 普通 rotate 使用 current camera up/right。
- 空白处基于当前屏幕深度样本解析中心。
- reference plane 作为第二优先级参与。
- label 不参与导航中心。

仍未实现或未完全对齐的部分：

- 没有惯性旋转。Onshape 有 inertia/velocity 路径，释放后可能继续按最后速度衰减旋转；当前实现 pointer up 后立即停止。
- 没有 constrained orbit / Alt 约束旋转。
- viewport depth sampling 还是固定步长和简单平均，没有中心权重、离群点过滤、深度聚类等高级采样质量策略。
- bounds fallback 使用当前简化规则，未完全复刻 Onshape 内部 `doBoundsFitInViewport` 相关细节。
- 没有 selected object / hovered object orbit center 策略。
- fit view / standard view 没有平滑动画。
- 没有更完整的 up-vector 极区稳定策略。
- 当前 renderer 已完成 WebGL2 基础迁移；长期专业 CAD 渲染内核可继续评估更完整的 WebGL2-native pick/depth buffer 或 WebGPU 路线。

### 后续建议顺序

1. 先保持当前相机基础行为稳定，不急于加惯性。
2. 若继续向 Onshape 靠，优先增强 depth sampling 质量：中心权重、离群点过滤、深度聚类、样本置信度。
3. 再考虑 WebGL2-native 的统一 pick/depth buffer 架构，把 navigation depth、selection picking、hover picking 收敛到同一套 render target 基础设施。
4. 之后再做 inertia rotation，且惯性期间必须继续围绕 pointerDown 时解析出的同一个 center。
5. 最后再做 constrained orbit、selected-object orbit center 和视图切换动画。
