## 当前产品方向

项目目标是三维云端 CAD 工作台，不再做二维出图编辑器。

核心技术方向已经确定：

- 页面：React
- 构建：Vite
- 包管理：pnpm workspace
- 项目 Node/pnpm 管理：mise，仅对当前项目生效
- 渲染：原生 WebGL，不引入 Three.js
- 几何内核：未来通过 `occt-draw-core` Wasm 提供
- 草图、约束、命令、选择、参数化编辑状态：TypeScript 实现
- 文档、产品文案和工程文档：中文

## 关键架构理解

### EditDraft

`EditDraft` 是命令执行过程中的临时编辑层。

它不是 UI，也不是 renderer。它用于承载：

- 鼠标移动过程中的临时线段。
- 拖动过程中的临时位置。
- 可取消的草图预览。
- 后续约束求解过程中的临时 working document。

标准规则：

- `pointerDown` 创建或初始化 `EditDraft`。
- `pointerMove` 更新 `EditDraft`。
- `pointerUp` 生成一个正式 `DocumentOperation`。
- `Esc` 丢弃 `EditDraft`。

### Operation

`DocumentOperation` 不是 Request。

- Request/Input：用户刚刚做了什么输入，例如鼠标移动。
- Command：当前命令如何解释输入。
- EditDraft：命令过程中的临时数据。
- Operation：最终写入 CAD 文档的正式业务动作。
- Transaction：一次原子提交。
- TransactionGroup：一个用户动作包含多次提交。

拖动 300 次不能产生 300 条历史。拖动过程更新 `EditDraft`，确认后只提交一次 `Operation` 或 `TransactionGroup`。

### Display

`display` 替代原来的 `scene`。

它的职责只有一个：把正式文档和临时编辑层转换为可渲染数据。

```txt
CadDocument + EditDraft -> DisplayModel -> renderer -> renderer-webgl
```

`display` 不知道 React、不知道 WebGL、不知道 CommandManager，也不修改 `CadDocument`。

### Editor

`editor` 是已落地的编辑器运行时包。

它不是 UI，也不是 CAD 文档模型。它用于承接 UI 之外的编辑器交互能力：

- 当前命令和命令生命周期。
- 鼠标、键盘、视图交互输入的解释。
- 选择状态和命令状态流转。
- `EditDraft` 的创建、更新和取消。
- `DocumentOperation / DocumentTransaction` 的提交入口。

目标是让 `apps/web` 只负责产品装配、界面展示和事件转发，避免命令逻辑继续散落在 React 组件或应用层控制器中。

## 当前不要做的事

- 不要回到二维出图方向。
- 不要先做复杂 UI。
- 不要引入 Three.js 或 gl-matrix。
- 不要把命令逻辑写进 React 组件。
- 不要让 renderer-webgl 反向知道 core、command 或 sketch。
- 不要在鼠标移动过程中频繁写正式文档历史。
- 不要为了演示效果绕过 `EditDraft / Operation / display` 边界。
- 不要默认开子线程或子 Agent，除非用户明确要求。
- 不要新增单元测试作为当前主线；当前用户更倾向浏览器手动验收，但 `pnpm check` 必须通过。

## 下一步推荐

下一步应开始 **草图系统 v1** 的最小闭环。

推荐实施顺序：

1. 定义默认草图平面和草图坐标系，第一版使用 XY 平面。
2. 定义最小草图数据模型：Sketch、SketchPoint、SketchLine。
3. 在 `editor` 中新增 `SketchLineCommand`。
4. 第一次点击记录起点到 `EditDraft`。
5. 鼠标移动时用 `EditDraft` 显示临时线段。
6. 第二次点击生成 `AddSketchLineOperation`。
7. `DocumentTransaction` 提交到 `CadDocument`。
8. `display` 把正式草图线投影为 `LineSegmentsDisplayObject`。
9. WebGL 显示正式草图线。

第一版不做：

- 圆、圆弧、多段线
- 尺寸标注
- 约束求解
- 拉伸实体
- Wasm 调用
- 云端保存

## 用户偏好和工作方式

- 要工业级产品思路，不接受临时偷懒方案。
- 代码和文档要能解释清楚架构意图。
- 文档用中文。
- UI 当前只保留必要入口，避免发散。
- 出方案时优先用架构图、流程图和简洁说明。
- 实现后要明确说明做了什么、是否完成之前确认的问题。
- 当前不希望默认使用多个子线程并行开发。
- 每次重要改动后至少跑 `pnpm check`。
